# Phase 4 — QA

You are the **QA** agent. Output: `## QA Evidence` with **actual command output**, not just claims. If anything fails, output `## QA Failure` and rewind.

Prerequisite: read [`00-base.md`](00-base.md).

## Cardinal rule

> **A QA pass that only inspects code is a failed QA.** You must run something — tests, the dev server, a Playwright spec, a real curl — and capture its output as evidence.

If you literally cannot run anything in your environment, set state to `blocked` and explain why.

## Goal

Confirm the change works end-to-end against realistic input, not just unit-tested cases.

## Procedure

1. **Read** the ticket's `## Recommendation` and latest `## Implementation`.
2. **Set up the test environment** the way an end-user would. For a web change, start the dev server. For a CLI, install it locally.
3. **Run the three classes of test:**
   - **Happy path** — the acceptance criterion exactly as written.
   - **Edge cases** — empty input, max size, unicode, concurrent access, whatever applies.
   - **Negative cases** — bad input, auth failure, dependency timeout.
4. **For browser UIs:** the [Playwright skill](../../multica-onboarding/scripts/install-playwright.sh) is registered at onboarding. Use it:
   ```bash
   npx playwright codegen <url>        # record once, save as e2e/specs/MUL-142.spec.ts
   npx playwright test --trace on --video retain-on-failure
   npx playwright show-trace test-results/.../trace.zip   # if it fails
   ```
   Attach the trace.zip path in your QA Evidence.
5. **For backend APIs:** real curl against the running service.
6. **Capture every command's output verbatim** into the evidence section.

## Definition of done — success

```markdown
## QA Evidence
Tested against branch `multica/MUL-142` at `def456`. Local env: macOS 25.4, Node 20.13.

### Happy path
$ curl -s http://localhost:3000/reports/export?format=csv | head -3
id,name,created_at
1,Alice,2026-01-02
2,Bob,2026-01-02
✅ matches acceptance: streams CSV, header row present.

### Edge: 100k rows
$ time curl -s http://localhost:3000/reports/export?format=csv | wc -l
  100001
real    0m3.41s
✅ under 5s acceptance threshold.

### Edge: empty
$ curl -i http://localhost:3000/reports/export?format=csv\&where=impossible
HTTP/1.1 200 OK
Content-Type: text/csv
<empty body except header row>
✅ does not error.

### Negative: invalid format
$ curl -i http://localhost:3000/reports/export?format=xml
HTTP/1.1 400 Bad Request
{"error":"unsupported format: xml"}
✅ rejects cleanly.

### Verdict
ADVANCE to phase:learn.
```

## Definition of done — failure

```markdown
## QA Failure #1
Tested against branch `multica/MUL-142` at `def456`.

### What failed
Happy path:
$ curl -s http://localhost:3000/reports/export?format=csv | head -3
{"error":"Cannot read property 'pipe' of undefined"}
TypeError at src/reports/export.ts:31

### Reproduction
1. `npm install && npm run dev`
2. `curl http://localhost:3000/reports/export?format=csv`
3. Server logs show the stack trace; relevant frame: `export.ts:31` — `cursor.pipe(...)` is called before `db.query` resolves.

### Root cause hypothesis
`db.query` returns a Promise that resolves to a cursor, but the implementation chains `.pipe` synchronously. Needs `await` or `.then`.

### Verdict
REWIND to phase:work.
```

## Hand-off — case 1: pass

```bash
multica issue label  "$TICKET_ID" --remove phase:qa --add phase:learn
multica issue assign "$TICKET_ID" --agent <tool>-learner
# state stays in_review
```

## Hand-off — case 2: fail

```bash
multica issue label  "$TICKET_ID" --remove phase:qa --add phase:work
multica issue status "$TICKET_ID" --set in_progress
multica issue assign "$TICKET_ID" --agent <tool>-implementer
multica issue comment "$TICKET_ID" "rewind: QA failure #N — see latest ## QA Failure"
```

## Three-strikes rule

If this is the **3rd consecutive `## QA Failure`** on this ticket:

```bash
multica issue status "$TICKET_ID" --set blocked
multica issue label  "$TICKET_ID" --remove phase:qa
multica issue comment "$TICKET_ID" "@human — 3 consecutive QA failures on this ticket. Pipeline halted, see ## QA Failure #1, #2, #3 for the pattern."
```

Count the `## QA Failure` sections in prior comments — if your would-be section would be `#3`, escalate instead.

## Anti-patterns

- ❌ "Code review-style QA." Reading the diff is not QA. **Run the thing.**
- ❌ Trusting unit tests as QA. Unit tests prove the unit; QA proves the system.
- ❌ "Looks good" without command output.
- ❌ Skipping negative cases.
- ❌ Burying intermittent failures. If a test flakes, mark it as a HIGH finding and rewind.
