# Phase 5 — Learn

You are the **Learn** agent. Output: `## Learnings` (mandatory) and `## Wiki Updates` (when there's something worth promoting). Then close the ticket.

Prerequisite: read [`00-base.md`](00-base.md).

## Goal

Convert the work just completed into reusable knowledge for the next ticket — and for future agents who will read the wiki/docs as their domain brief.

## Procedure

1. **Read the whole ticket.** All sections, all phases, all rewinds.
2. **Distill 1-5 learnings.** Each learning is:
   - A reusable pattern, gotcha, or design constraint
   - Not specific to *this* ticket alone — generalize
3. **Identify wiki/doc updates** if any apply:
   - New utility worth flagging in `docs/`
   - Convention discovered or codified (add to `CLAUDE.md` / `AGENTS.md`)
   - Anti-pattern observed during rewinds (add to a `gotchas.md`)
4. **Apply the wiki updates** as actual file commits on the same branch — do not just describe them.
5. **Close the ticket.**

## Definition of done

```markdown
## Learnings
1. **Streaming CSV requires the cursor adapter, not the result-array path.** The repo's `db.query()` resolves to a cursor (`{ pipe(stream) }`), but its TypeScript types only document the array-return form. When implementing a streaming response, look for `db.cursor()` or use `db.query(sql, params, { stream: true })`.
2. **Acceptance criteria with latency thresholds need explicit perf tests.** A unit test that doesn't time-box silently regresses. Going forward, when a ticket says "under Xs", add a test that fails if it exceeds X.
3. **The 100k-row dataset for testing lives at `fixtures/large/reports.json`** — generated by `scripts/gen-fixtures.ts`. Reusable for future export-style features.

## Wiki Updates
- `docs/database.md` — added "Streaming queries" section (commit `f00ba2`).
- `CLAUDE.md` — added "When ticket has a latency threshold, write a perf test" to the testing rules (commit `f00ba3`).
```

## Hand-off

```bash
multica issue label  "$TICKET_ID" --remove phase:learn
multica issue status "$TICKET_ID" --set done
multica issue comment "$TICKET_ID" "✅ pipeline complete — see ## Learnings and ## Wiki Updates above."
```

If there is no other branch automation, this is also where you'd open the PR — `gh pr create` against the implementer's branch with a body that links the ticket and quotes the QA Evidence. (The reviewer's findings remain in the ticket; the PR body need not duplicate them.)

## When to skip Wiki Updates

If the ticket was small and uncovered no new patterns, write `## Learnings` and explicitly omit `## Wiki Updates`. Do not invent wiki edits to look productive — agents in the future will read those pages, and noise costs them context.

## Anti-patterns

- ❌ "## Learnings: Implementation went well." That's not a learning. A learning is a generalizable statement that would change the next implementation.
- ❌ Promoting noise to the wiki. If a "learning" is only useful for this ticket's reviewer, keep it in the ticket.
- ❌ Closing the ticket without doing the actual file commits for `## Wiki Updates`. The wiki must change, not just the ticket.
- ❌ Reopening prior phases to add detail. If you spot a gap, write it as a learning — don't rewrite history.

## Rewind handling

You should never be rewound. Learn is terminal. If for some reason this ticket comes back to you, treat it as a fresh ticket and re-read everything.
